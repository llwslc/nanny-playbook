# nanny-playbook

一份**独立自洽**的育儿嫂招聘与管理指南。公开仓库，**不放任何个人信息**。

---

## 写作铁律：指南不引用读者看不见的东西

**这份文档的读者，手上只有这份文档——很可能就是一沓打印出来的纸。**

他没参与过任何讨论，没看过任何草稿，没读过任何被拿来对照的外部材料，也没问过任何问题。
**他也没有这个仓库**：不用 git，看不到 `本地/`，不知道什么叫"提交"。

**所以只做正面陈述。** 以下都是 bug：

| ❌ 不要写 | 为什么是 bug |
|---|---|
| 反驳一个指南从未提出的主张<br>（"先扔掉一个错误指标 X"、"别用 X 那套"） | 读者没见过 X。这句话对他是**悬空**的 |
| 假设读者的职业背景或心理活动<br>（"如果你在公司做 OKR，会很自然地想……"） | 读者不一定是那种人 |
| 把某次讨论的**答复**原样搬进来 | 那是答复，不是指南 |
| 引用外部材料<br>（"你发我的那份套餐"、"上一版"） | 读者看不见它 |
| 叙述者第一人称（"我建议""我觉得"） | 指南没有"我" |
| 提到仓库、git、目录结构<br>（"不要提交到公开仓库"、"放 `本地/`"、"改源 md 会自动重建"） | 他拿的是打印稿。这些字印在纸上没人看得懂 |

### 写给仓库使用者的话，包进 skip 标记

git、`.gitignore`、"别提交填好的表"——这些**该写**，但只对着仓库看才成立。包起来，它们就不进合订本：

```markdown
<!-- onepage:skip-start -->
## 仓库结构
...
<!-- onepage:skip-end -->
```

给维护者的话（"本文件自动生成，不要手改"）写进 **HTML 注释**——注释不渲染，也就不进 PDF。

### 正确的做法

**从讨论、纠错、外部材料里学到的东西，要以「正面结论」落盘，不能以「对它的反驳」落盘。**

举个真实的例子——审一份体检套餐时发现它拿"便常规"冒充"粪便培养"：

- ❌ 写成 **"套餐里的便常规不算数"** → 读者没见过那份套餐，看不懂
- ✅ 写成 **清单里加一列「验收：报告上必须出现『粪便培养』；写『便常规』不算」**

后者不依赖任何一份套餐，**反过来还能用来验收任意一份套餐**。

**输入的形状，不该出现在产出里。**

`node tools/lint-docs.mjs` 检查这条。pre-commit 自动跑，违反则拒绝提交。

## 改了任何 md → 重建打印合订本

```bash
node tools/lint-docs.mjs       # 查对话残留 + 仓库残留
node tools/build-onepage.mjs   # → ONEPAGE.md + ONEPAGE.html
```

pre-commit 会自动跑（lint → build → 加入提交），任一步红灯拒绝提交。

**新增 md 必须登记进 `tools/onepage.order`**，否则构建失败——不允许被静默漏出打印版。

### 脚本放 `tools/`，不放 `.claude/skills/`

**因为 git 钩子要调它们。** 别人 clone 这个仓库、从不装 Claude，`git commit` 也得能跑——
项目不该依赖 Claude 的配置目录。

（skill 可以装脚本，这不是能力限制，是架构选择。）
