# nanny-playbook

一份**独立自洽**的育儿嫂招聘与管理指南。公开仓库，**不放任何个人信息**。

---

## 写作铁律：指南不引用读者看不见的东西

**这份文档的读者，手上只有这份文档——很可能就是一沓打印出来的纸。**

他没参与过任何讨论，没看过任何草稿，没读过任何被拿来对照的外部材料，也没问过任何问题。
**他也没有这个仓库**：不用 git，看不到 `本地/`，不知道什么叫"提交"。

**所以只做正面陈述。** 以下都是 bug：

| ❌ 不要写 | 为什么是 bug |
|---|---|
| 反驳一个指南从未提出的主张<br>（"先扔掉一个错误指标 X"、"别用 X 那套"） | 读者没见过 X。这句话对他是**悬空**的 |
| 假设读者的职业背景或心理活动<br>（"如果你在公司做 OKR，会很自然地想……"） | 读者不一定是那种人 |
| 把某次讨论的**答复**原样搬进来 | 那是答复，不是指南 |
| 引用外部材料<br>（"你发我的那份套餐"、"上一版"） | 读者看不见它 |
| 叙述者第一人称（"我建议""我觉得"） | 指南没有"我" |
| 提到仓库、git、目录结构<br>（"不要提交到公开仓库"、"放 `本地/`"、"改源 md 会自动重建"） | 他拿的是打印稿。这些字印在纸上没人看得懂 |

### 写给仓库使用者的话，包进 skip 标记

git、`.gitignore`、"别提交填好的表"——这些**该写**，但只对着仓库看才成立。包起来，它们就不进合订本：

```markdown
<!-- onepage:skip-start -->
## 仓库结构
...
<!-- onepage:skip-end -->
```

给维护者的话（"本文件自动生成，不要手改"）写进 **HTML 注释**——注释不渲染，也就不进 PDF。

### 正确的做法

**从讨论、纠错、外部材料里学到的东西，要以「正面结论」落盘，不能以「对它的反驳」落盘。**

举个真实的例子——审一份体检套餐时发现它拿"便常规"冒充"粪便培养"：

- ❌ 写成 **"套餐里的便常规不算数"** → 读者没见过那份套餐，看不懂
- ✅ 写成 **清单里加一列「验收：报告上必须出现『粪便培养』；写『便常规』不算」**

后者不依赖任何一份套餐，**反过来还能用来验收任意一份套餐**。

**输入的形状，不该出现在产出里。**

`node tools/lint-docs.mjs` 检查这条。pre-commit 自动跑，违反则拒绝提交。

## 改了任何 md → 重建打印合订本

```bash
node tools/lint-docs.mjs       # 查对话残留 + 仓库残留
node tools/build-onepage.mjs   # → ONEPAGE.md + ONEPAGE.html
```

pre-commit 会自动跑（lint → build → 加入提交），任一步红灯拒绝提交。

**新增 md 必须登记进 `tools/onepage.order`**，否则构建失败——不允许被静默漏出打印版。

### 脚本放 `tools/`，不放 `.claude/skills/`

**因为 git 钩子要调它们。** 别人 clone 这个仓库、从不装 Claude，`git commit` 也得能跑——
项目不该依赖 Claude 的配置目录。

（skill 可以装脚本，这不是能力限制，是架构选择。）

## 跨文件引用：按标题引，别按编号引

### 章节引用 `§xxx` —— 只能按标题

```
❌  §九「孩子几个月能睡整觉？」      按编号
✅  §睡眠「孩子几个月能睡整觉？」    按标题
```

**为什么编号不行**：往前面插一章，后面的编号**全部平移**。那个引用**依然在范围内、依然"合法"**，
但已经指向另一章了。文档还是通顺的，只是每一处都在指错东西——**没有任何东西会报错。**

> ⚠️ **这个 bug 真出过，而且是在门禁眼皮底下出的。**
>
> 插入「大便与便秘」「黄疸」两章之后，`§九` 从「睡眠」变成了「黄疸」，
> 而 `危险信号.md` 还写着"基线 → §九 孩子几个月能睡整觉"。
> 当时的门禁只查「§N 在不在范围内」——**九在范围内，照样绿灯。**
>
> **教训：范围检查抓不住「改了意思但没越界」。**

**按标题引免疫这一切**：章节怎么重排、怎么改号，`§睡眠` 都指得着。

`lint-docs` 强制这条：**写 `§数字` 直接报错**；写 `§标题` 会去目标文件里核对标题在不在——**改名了也会红**。
（目标文件 = 该行 `§` 之前最后一个 `.md` 链接；没有链接就是引自己。）

### 红线条号 `红线第 N 条` —— 只能追加

红线是**签字生效的合同附件**，条号改不成标题引。所以规则更硬：

**只能追加到末尾。不能插队、不能重排、不能删。**

> ⚠️ **门禁在这里有盲区，必须知道：**
>
> 它查得了 **越界**（引用第 99 条但只有 14 条）、**断号**（1,2,4,5）、**两张表对不上**。
> 但它**查不了**「在第 4 条前面插一条、把原来的第 12 条挤成第 13 条」——
> 编号依然连续、引用依然在范围内，**照样绿灯，照样静默指错**。
>
> **所以「只能追加」这条，得靠人守。**

### 动一条红线，要同时动三处

1. **对照表**（带「为什么」那列）
2. **签字页**（要打印、要签字的那一张）
3. **条数**：标题的 `## N 条` + `README.md` 目录说明里的那个数

> **签字页是法律上生效的那一版。**
> 一条规则在对照表里写得再狠，签字页上写弱了，签下去的就是弱的那一版。
> （真出过：对照表要求"每次睡觉都仰卧"，签字页却写"3 个月内不趴睡"——反过来读等于"3 个月后可以趴睡"。）

## 一级标题 = 合订本的章节名。两边冲突时，用 onepage.order 覆盖

`build-onepage.mjs` 默认取每个文件**自己的 `# 一级标题`** 当章节名，印在目录里、印在章首。

**但有的 H1 只对着仓库成立。** `README.md` 的 H1 是 `# nanny-playbook`——
在 GitHub 上它**就该**是仓库名；印到打印稿上却是天书，读者不知道这东西在网上叫什么。

**这种情况不要改源文件**（改了 GitHub 那边就没名字了）。在 `tools/onepage.order` 里覆盖：

```
README.md | 怎么用这份指南
```

`文件 | 章节标题` —— build 会把**目录和章首的 H1 一起**换成覆盖的标题，源文件一个字不动。

lint 配套：打印区里出现仓库名 = 红灯，**但被覆盖掉的那个 H1 豁免**（它印不到纸上）。
其他任何位置泄漏仓库名，照样红灯。
